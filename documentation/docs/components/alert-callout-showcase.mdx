## Real-World Documentation Examples

<Callout variant="info" title="About Storage Types">
  Soroban provides three types of storage:
  
  - **Temporary**: Deleted after a short period, cheapest option
  - **Persistent**: Remains until explicitly deleted, moderate cost
  - **Instance**: Tied to contract instance lifetime, balanced cost
  
  Choose the appropriate type based on your data lifecycle requirements.
</Callout>

<Callout variant="tip" title="Gas Optimization Strategy">
  To minimize gas costs in your Soroban contracts:
  
  1. Use <code>Temporary</code> storage for ephemeral data
  2. Batch operations when possible to reduce transaction overhead
  3. Implement early returns to avoid unnecessary computation
  4. Cache frequently accessed values in local variables
  5. Use efficient data structures (e.g., <code>Map</code> over <code>Vec</code> for lookups)
</Callout>

<Callout variant="warning" title="Common Pitfall">
  When working with timestamps in Soroban, remember that <code>env.ledger().timestamp()</code> returns the ledger close time, not the current wall clock time. This value only updates when a new ledger is closed (approximately every 5 seconds).
</Callout>

<Callout variant="error" title="Anti-Pattern: Unbounded Loops">
  Avoid unbounded loops in your contract code:
  
  ```rust
  // ❌ BAD: Unbounded loop
  while condition {
      // This could run forever and exceed gas limits
  }
  
  // ✅ GOOD: Bounded iteration
  for i in 0..MAX_ITERATIONS {
      if !condition { break; }
      // Safe iteration with upper bound
  }
  ```
  
  Always set explicit limits to prevent gas exhaustion and ensure contract reliability.
</Callout>

<Callout variant="success" title="Best Practice: Input Validation">
  Always validate input parameters at the beginning of your contract functions:
  
  ```rust
  pub fn transfer(env: Env, from: Address, to: Address, amount: i128) {
      // ✅ Validate inputs first
      from.require_auth();
      
      if amount <= 0 {
          panic!("Amount must be positive");
      }
      
      if from == to {
          panic!("Cannot transfer to self");
      }
      
      // Now proceed with business logic
      // ...
  }
  ```
  
  This pattern provides clear error messages and prevents wasted computation on invalid inputs.
</Callout>
